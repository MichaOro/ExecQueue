## Anforderungsdokument

**Version**: 1.0  
**Datum**: 22.04.2026  
**Status**: Entwurf

---

## 1. Zielsetzung

Die Telegram-Integration ermöglicht die Remote-Steuerung von ExecQueue über einen Chat-Bot. Benutzer können Aufgaben anstoßen, Status abfragen und das System kontrollieren, ohne direkten Zugriff auf die API oder Datenbank zu benötigen.

---

## 2. Kontext

ExecQueue orchestriert Aufgaben als Tasks mit automatischer Ausführung über OpenCode. Der Telegram-Bot dient als primäre Schnittstelle für:
- **Operatoren**: Starten/Stoppen von Tasks, Status-Abfragen
- **Administratoren**: System-Health, Konfiguration
- **Beobachter**: Status-Updates, Ergebnis-Benachrichtigungen

---

## 3. Architektur-Grundlagen

Basierend auf ExecQueue-Architektur:
- User (Telegram) → ExecQueue API (FastAPI) → Database → Scheduler/Runner → OpenCode
- Task-Lifecycle: `queued → in_progress → validation → done/retry/failed`
- Validierung via JSON-Output: `{ "status": "done" | "not_done", "summary": "...", "evidence": "..." }`

## 5. Nicht-funktionale Anforderungen

### 5.1 Sicherheit

| ID | Anforderung |
|----|-------------|
| NF-01 | Telegram-Bot-Token muss in `.env` gespeichert werden (`TELEGRAM_BOT_TOKEN`) |
| NF-02 | Nur autorisierte Benutzer dürfen Commands ausführen |
| NF-03 | Admin-Commands erfordern explizite Berechtigung |
| NF-04 | Sensible Daten (API-Keys, Credentials) dürfen nicht in Bot-Nachrichten erscheinen |
| NF-05 | Rate-Limiting pro Benutzer zur Vermeidung von Missbrauch |

### 5.2 Performance

| ID | Anforderung |
|----|-------------|
| NF-10 | Response-Zeit für Commands < 2 Sekunden |
| NF-11 | Support für bis zu 100 gleichzeitige Benutzer |
| NF-12 | Queue-Übersicht auf 10 Einträge begrenzt (Pagination optional) |

### 5.3 Zuverlässigkeit

| ID | Anforderung |
|----|-------------|
| NF-20 | Bot muss bei API-Ausfall gracefully degradieren (Error-Nachricht) |
| NF-21 | Nachrichten müssen genau einmal verarbeitet werden |
| NF-22 | Keine Datenverluste bei Bot-Neustart |

### 5.4 Benutzererfahrung

| ID | Anforderung |
|----|-------------|
| NF-30 | Klare, prägnante Nachrichten ohne technisches Jargon |
| NF-31 | Inline-Keyboards für häufige Aktionen (z.B. Confirm/Cancel) |
| NF-32 | Command-Hilfe via `/help` muss alle Commands auflisten |
| NF-33 | Markdown-Formatierung für bessere Lesbarkeit |

## 7. Schnittstellen

### 7.1 Telegram Bot API

- **Webhook**-Modus für Produktion (empfohlen)
- **Polling**-Modus für Entwicklung
- Supported API-Version: Bot API 6.0+

### 7.2 Interne API-Endpunkte (FastAPI)

Der Bot konsumiert bestehende ExecQueue-API-Endpoints:
- `GET /api/tasks` - Task-Liste
- `GET /api/tasks/{task_id}` - Task-Detail
- `POST /api/tasks` - Neuer Task
- `PATCH /api/tasks/{task_id}/cancel` - Task abbrechen
- `POST /api/tasks/{task_id}/retry` - Retry
- `GET /api/requirements` - Requirements
- `GET /api/health` - System-Health
- `POST /api/scheduler/start` - Scheduler starten
- `POST /api/scheduler/stop` - Scheduler stoppen

## 9. Rollen & Berechtigungen

| Rolle | Commands | Beschreibung |
|-------|----------|--------------|
| **observer** | `/start`, `/help`, `/queue`, `/status`, `/health`, `/tasks` | Nur-Lese-Zugriff |
| **operator** | observer + `/create`, `/cancel`, `/retry` | Task-Operations |
| **admin** | operator + `/start_scheduler`, `/stop_scheduler`, `/admin *` | Vollzugriff |

## 10. Fehlerbehandlung

| Szenario | Verhalten |
|----------|-----------|
| API nicht erreichbar | Bot meldet "Service temporarily unavailable" |
| Ungültiger Task-ID | Bot meldet "Task not found" mit Format-Hilfe |
| Nicht-autorisiert | Bot meldet "Access denied" ohne Details |
| Invalid-Prompt | Bot fordert korrektes Format an |
| Scheduler bereits aktiv/inaktiv | Bot bestätigt aktuellen Status |

ADMIN_USER_IDS=123456789,987654321
RATE_LIMIT_PER_MINUTE=30
