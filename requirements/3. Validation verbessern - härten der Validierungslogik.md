# Anforderungsdokument: Validation Hardening

## 1. Zielsetzung

Härtung der Validierungslogik in ExecQueue zur Sicherstellung zuverlässiger, deterministischer und fehlertoleranter Task-Validierung.

---

## 2. Current State (Baseline)

Aus `execqueue_architektur.md`:

- **Validation Status**: basic
- **Output Format**: JSON Pflicht `{ "status": "done" | "not_done", "summary": "...", "evidence": "..." }`
- **Task Lifecycle**: queued → in_progress → validation → done | retry | failed
- **Retry-Logik**: retry_count, max_retries vorhanden

---

## 3. Anforderungen

### 3.1 Formale Validierung

**REQ-VAL-001: JSON Schema Enforcement**
- Validierung muss striktes JSON Schema prüfen
- Abweichungen führen zu automatischem Retry mit Fehlermeldung
- Schema muss versionierbar sein

**REQ-VAL-002: Output Parsing Robustheit**
- Handling von malformed JSON (try-catch mit Recovery)
- Support für JSON innerhalb von Markdown Code-Blöcken
- Timeout bei parsing-blockierenden Eingaben

### 3.2 Semantische Validierung

**REQ-VAL-003: Status-Konsistenz**
- `status: done` erfordert nicht-leeren `evidence`-String
- `status: not_done` erfordert Begründung in `summary`
- Cross-field Validierung vor Status-Akzeptanz

**REQ-VAL-004: Evidence Quality**
- Evidence muss konkrete Referenzen enthalten (Dateinamen, Zeilennummern, Test-Outputs)
- Minimum length requirement für evidence (configurable)
- Pattern-Matching für kritische Artefakte (z.B. "test passed", "no errors")

### 3.3 Retry-Strategie

**REQ-VAL-005: Differenzierte Retry-Logik**
- Differentiation zwischen:
  - Parsing-Fehlern (sofortiger Retry mit Parsing-Hilfe)
  - Semantic-Fehlern (Retry mit verbessertem Prompt)
  - Critical-Failures (kein Retry, direkter Failed-Status)

**REQ-VAL-006: Exponential Backoff**
- Retry-Intervalle mit exponentieller Verzögerung
- Maximal 3 Retries ohne manuelle Intervention
- Jitter zur Vermeidung von Thundering Herd

**REQ-VAL-007: Retry Context Preservation**
- Vorherige Validation-Fehler müssen im Prompt verfügbar sein
- Accumulation von Failure-Evidence für Debugging
- Retry-History im Task-Record

### 3.4 Verifikationsmechanismen

**REQ-VAL-008: Multi-Pass Validation**
- Erstpass: Strukturierte Validierung (JSON + Schema)
- Zweipass: Semantische Prüfung (Inhaltsqualität)
- Drittpass: Cross-Reference mit Requirements

**REQ-VAL-009: Deterministic Checks**
- Unit-Test Execution vor Acceptance
- Linting/Static Analysis als Validation-Step
- Build-Status Verification bei compilable Artefakten

**REQ-VAL-010: Human-in-the-Loop Escalation**
- Nach N fehlgeschlagenen Retries: Manuelle Review Queue
- Escalation-Threshold konfigurierbar
- Notification an Stakeholder bei Escalation

### 3.5 Monitoring & Observability

**REQ-VAL-011: Validation Metrics**
- Tracking von:
  - Validation Success Rate
  - Average Retries per Task
  - Common Failure Patterns
  - Mean Time to Validation

**REQ-VAL-012: Audit Trail**
- Jede Validierung muss loggable sein
- Full Request/Response History
- Immutable Validation Records für Compliance

### 3.6 Konfigurierbarkeit

**REQ-VAL-013: Policy-Based Validation**
- Differentierte Policies pro Task-Typ
- Environment-specific Thresholds (dev vs. prod)
- Hot-reload von Validation Rules

**REQ-VAL-014: Graceful Degradation**
- Fallback-Policies bei Service-Ausfällen
- Configurable Strictness Levels
- Circuit Breaker für External Validation Services

---

## 4. Nicht-Funktionale Anforderungen

### 4.1 Performance

**NFR-VAL-001**: Validation-Durchlauf < 5 Sekunden (ohne External Calls)
**NFR-VAL-002**: Parallel Validation von bis to 10 Tasks gleichzeitig

### 4.2 Sicherheit

**NFR-VAL-003**: Kein Code Execution in Validation-Phase
**NFR-VAL-004**: Input Sanitization für alle External Prompts

### 4.3 Wartbarkeit

**NFR-VAL-005**: Validation Rules als deklarative Konfiguration
**NFR-VAL-006**: Unit-Test Coverage > 90% für Validation-Logik

---

## 5. Akzeptanzkriterien

1. **Schema-Validierung**: 100% der Validation-Outputs werden auf JSON Schema geprüft
2. **Retry-Dokumentation**: Jeder Retry hat dokumentierten Grund
3. **Escalation**: Alle Tasks nach 3 Retries eskalieren korrekt
4. **Metrics**: Validation-Metrics sind in Echtzeit abrufbar
5. **Zero Regression**: Bestehende Validierungsflüsse bleiben kompatibel
